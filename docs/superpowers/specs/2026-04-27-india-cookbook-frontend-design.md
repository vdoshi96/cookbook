# India Cookbook Frontend Design

## Goal

Build the first version of the India Cookbook website as a quiet, bookish, searchable frontend for the static JSON in `/data/`. The site should feel like an editorial companion to the cookbook, not just a recipe database: chapters, region context, front matter, cross-references, and ingredients all need to invite browsing.

## Approved Direction

The website should use a **bookish editorial** experience model with a **chapters-first** home page. The visual system should be **quiet literary**: readable, warm, restrained, and suitable for long browsing sessions on both laptop and mobile.

Core decisions from the design interview:

- The home page leads with chapters, while keeping search prominent.
- The base look is quiet and readable, with restrained editorial styling.
- Dish and section imagery should be curated from the web in the frontend layer, not taken from scanned PDF pages generated by the backend.
- The site must support both light and dark mode as first-class themes.
- Recipe pages should be compact and practical, with the image flowing out of the way as the user scrolls.
- Search should include both instant suggestions and a dedicated `/search` page.
- Filtering should use a persistent desktop sidebar and a mobile slide-in side sheet similar to flight-search websites.
- UI copy should be blended: clear and neutral for controls, warmer and more editorial for intros, rails, and empty states.

## Experience Principles

1. **The book structure leads.** Chapters are the primary home-page doorway because the source material is organized around them.
2. **Search is always close.** Search should feel fast and useful without turning the whole site into a generic index.
3. **Exploration is built from relationships.** Regions, ingredients, tags, cross-references, "used in", and related rails should make wandering natural.
4. **Recipe pages serve cooking first.** Photos and context matter, but time, heat, serves, ingredients, and method must be quickly reachable.
5. **Mobile and laptop are equal priorities.** Layouts should adapt intentionally instead of merely collapsing desktop designs.
6. **Theme support is designed.** Dark mode should use warm charcoal surfaces and readable contrast, not a naive inversion of the light palette.

## Visual System

### Tone

The visual language is quiet literary:

- Warm paper background in light mode.
- Warm charcoal background in dark mode.
- Serif-led editorial headings.
- Clear sans-serif UI labels where readability benefits.
- Thin borders, compact surfaces, and modest corner radii.
- No heavy decorative frames, no loud gradients, and no stock-looking decorative filler.

### Light Theme

Use a warm paper palette:

- Page background: warm paper.
- Surface background: slightly lighter paper.
- Text: near-black warm ink.
- Muted text: warm gray-brown.
- Accent: restrained brown or copper.
- Borders: low-contrast warm ink.

### Dark Theme

Use a warm charcoal palette:

- Page background: deep warm charcoal.
- Surface background: slightly raised charcoal.
- Text: warm cream.
- Muted text: softened cream.
- Accent: muted copper.
- Borders: low-contrast cream.

Images in dark mode should receive subtle overlays or filters so bright food photos do not glare against the page.

### Theme Behavior

The app should:

- Respect the user's system preference on first visit.
- Provide a visible light/dark theme toggle.
- Persist the selected theme locally.
- Avoid hydration flicker where possible.

## Image Strategy

The backend `image` field is optional and should not be treated as the preferred visual source for v1 because generated images may come from scanned PDF pages. The frontend should own a curated image mapping for dishes, sections, and regions.

Implementation guidance:

- Store curated image metadata under `/web`, not `/data`.
- Use recipe IDs, section IDs, and region IDs as lookup keys.
- Include image URL, alt text, and optional source label.
- Fall back gracefully to a quiet typographic placeholder when no curated image exists.
- Keep images responsive and avoid layout shifts with fixed aspect ratios.

The user is not requiring license-safe sources for this personal-use site, but the implementation should keep source metadata possible so the policy can change later without restructuring.

## Information Architecture

### Home

The home page is a handsome table of contents.

Required content:

- Site title and primary navigation.
- Prominent search input with instant suggestions.
- Chapter grid using `sections.json`.
- Short chapter intro snippets.
- A "from the introduction" or front-matter rail using `front-matter.json`.
- Optional region/ingredient invitation lower on the page.

Primary goal: encourage chapter browsing while preserving fast lookup.

### Section Page: `/sections/[id]`

Required content:

- Chapter title.
- Intro prose from `intro_markdown`.
- Start-here rail from `start_here`.
- Filterable recipe list for the section.
- Filters for region, time, heat, dietary tags, and technique tags.

Laptop layout:

- Intro and start-here content at the top.
- Filter sidebar next to recipe grid/list.

Mobile layout:

- Intro stack first.
- Active filters summary above results.
- Slide-in filter side sheet for controls.

### Region Page: `/regions/[id]`

Required content:

- Region title and intro prose.
- Map or coordinate-based visual treatment.
- Recipes from the region.
- Filter controls using the same responsive filter pattern.

The region visual can begin as a quiet map-card placeholder using `map_coords`; a richer India map can be added later.

### Recipe Page: `/recipes/[id]`

Required content:

- Recipe title and subtitle.
- Section and region links.
- Opening image band using curated image mapping when available.
- Prep time, cook time, serves, heat level, and notes.
- Ingredients.
- Method steps.
- Cross-references from `cross_refs`.
- Reverse edges from `graph.json` `used_in`.
- Related rails for more from the same region and section.

Layout behavior:

- The image is an opening band, not a sticky hero.
- The image scrolls away normally.
- Metadata appears immediately below the image.
- Laptop uses an efficient ingredients/method layout.
- Mobile uses a single clean stack.
- A compact sticky section navigation can link to ingredients, method, cross-references, and related recipes.

### Ingredient Page: `/ingredients/[name]`

Required content:

- Ingredient display name and count.
- Recipes that use it.
- Links back to related recipes.
- Same filter/result-card system where useful.

### About Page: `/about`

Required content:

- Front-matter sections from `front-matter.json`.
- Introduction, history, Ayurveda, regions overview, and notes on recipes.
- Markdown rendering with the same quiet literary reading style.

### Search Page: `/search`

Required content:

- Search input.
- Fuzzy client-side results.
- Filters for result type, region, section, time, heat, dietary, technique, and occasion where data exists.
- Result cards for recipes, ingredients, regions, sections, and tags.

Search should index:

- Recipe name and subtitle.
- Ingredients.
- Section name.
- Region name.
- Dietary, technique, and occasion tags.
- Cross-reference names where available.

Instant suggestions should be available from the header and home search. The dedicated page should support deeper comparison and filtering.

## Navigation

Primary navigation:

- Home.
- Chapters.
- Regions.
- Search.
- About.

The header should remain compact. On mobile, use a simple menu button and avoid crowding the search surface.

## Filters

Use one filtering model across section, region, ingredient, and search pages.

Laptop:

- A quiet left sidebar with grouped filters.
- Results remain visible while filters change.

Mobile:

- A filter button opens a slide-in side sheet from the side, similar to flight-search websites.
- Active filters are summarized above the result list.
- The side sheet should be easy to dismiss and should not hide the current context permanently.

Filter state should be represented in query parameters where practical so filtered views are shareable and navigable.

## Copy Voice

Use blended copy.

Neutral controls:

- "Search recipes, ingredients, regions"
- "Filter"
- "Clear filters"
- "View recipe"

Warmer context:

- "Start with these recipes"
- "More from Awadh"
- "Used in these dishes"
- "No matches yet. Try a region, ingredient, or technique."

Book-provided prose should carry most of the editorial voice. The UI should not over-perform personality in buttons or controls.

## Data Use

The frontend consumes static JSON from `/data/`:

- `recipes.json`
- `sections.json`
- `regions.json`
- `ingredients.json`
- `tags.json`
- `graph.json`
- `front-matter.json`

The frontend must not modify `/data/`. If a data contract change is needed, document the request for the user to coordinate with the backend workstream.

Use TypeScript types and validation-friendly data access helpers under `/web` so route components do not import raw JSON everywhere.

## Technical Direction

Build under `/web` with:

- Next.js App Router.
- TypeScript.
- Static routes using `generateStaticParams`.
- Static JSON imports or build-time data helpers.
- Client-side Fuse.js search.
- No runtime backend.
- No database.
- No auth.
- No server actions unless explicitly requested later.

Suggested frontend structure:

```text
/web
  /app
    /(site)
      page.tsx
      layout.tsx
      about/page.tsx
      search/page.tsx
      sections/[id]/page.tsx
      regions/[id]/page.tsx
      recipes/[id]/page.tsx
      ingredients/[name]/page.tsx
  /components
    Header.tsx
    SearchBox.tsx
    RecipeCard.tsx
    FilterSidebar.tsx
    FilterSideSheet.tsx
    RecipeImage.tsx
    ThemeToggle.tsx
  /lib
    data.ts
    search.ts
    filters.ts
    curated-images.ts
    types.ts
  /styles
    globals.css
```

Exact structure can be adjusted to match the scaffolded Next.js app, but responsibilities should stay clear.

## Accessibility And Responsiveness

Requirements:

- Keyboard-accessible search, filters, theme toggle, and mobile navigation.
- Accessible labels for icon buttons.
- Proper alt text for curated images.
- Good contrast in both themes.
- No text overlap on narrow screens.
- Stable card and image dimensions to prevent layout shifts.
- Touch-friendly controls on mobile.

## Performance

Requirements:

- Static generation for recipe, section, region, and ingredient routes.
- Search should respond in under 100ms on a recent laptop for the full data set.
- Avoid importing the full search index into every route if it creates noticeable bundle cost.
- Use responsive image sizing and lazy loading where appropriate.
- Keep the first home page render lightweight despite the ~1000-recipe corpus.

## Testing And Verification

Implementation should include:

- Type checking.
- Linting.
- Unit tests for data helpers and filtering/search behavior where the scaffold supports it.
- Browser verification for home, recipe, section, search, and theme switching.
- Desktop and mobile viewport checks.
- Dark and light mode checks.

## Non-Goals For V1

- Recipe scaling or unit conversion.
- User accounts.
- Saving favorites.
- Runtime database or backend API.
- Editing recipes.
- Full interactive India map beyond a simple region visual treatment.
- Guaranteed complete curated images for every recipe on day one.

## Open Implementation Notes

These are choices for the implementation plan, not unresolved design questions:

- Choose the exact Next.js scaffold command based on the current package manager defaults.
- Decide whether to use a small Markdown renderer or simple Markdown-to-HTML handling for the front matter.
- Decide whether to prebuild a compact search index file or generate it from static imports at build time.
